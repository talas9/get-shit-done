Key frontmatter points:

- No `# hooks:` block needed — this is a read-only agent (no Write tool)

**Body structure — include these sections in this order:**

1. `<role>` — L2 sub-orchestrator identity. Coordinates L3 agents, never executes work directly. Spawned by `/gsd:execute-phase` L1 orchestrator. State that this agent MUST NOT attempt TeamCreate (bug #32731 — standalone subagent only).

2. `<worktree_context>` — Bug #27749 workaround. Instruct agent to extract worktree path from `<worktree>` tag in the prompt. Use this as absolute base for ALL file paths. Never use `.` or relative paths. Example: `{worktree_path}/.planning/phases/...`

3. `<stream_context>` — Agent receives its assigned stream info from L1 prompt via `<stream>` tag containing JSON: `{ "name": "stream-a", "plans": ["02-01-PLAN.md", "02-02-PLAN.md"] }`. Also receives `<phase>` and `<phase_dir>` tags from L1.

4. `<execution_flow>` — Sequential plan execution loop:
   - For each plan in the stream's `plans` array (in order):
     a. Construct full plan path: `{worktree_path}/.planning/phases/{phase_dir}/{plan_filename}`
     b. Read the plan file to extract phase number and plan number
     c. Spawn L3 executor via Agent tool with `subagent_type="gsd-executor"`. In the prompt, include: the plan path, the phase identifier, and a `<worktree>` tag passing through the worktree path so L3 also uses absolute paths
     d. Do NOT use `run_in_background: true` — process plans sequentially within the stream
     e. After L3 returns, verify completion (see completion_check)
     f. If any L3 fails, stop immediately and write `STREAM_FAILED.md`

5. `<completion_check>` — AGNT-04: Validate L3 completion by file existence only.
   - After each L3 spawn returns, use Read tool to check if `{worktree_path}/.planning/phases/{phase_dir}/{phase}-{plan}-SUMMARY.md` exists
   - If Read succeeds (file exists) -> plan complete, continue to next
   - If Read fails (file not found) -> L3 failed, write STREAM_FAILED.md and stop
   - Do NOT parse or process SUMMARY.md content — existence is the signal

6. `<sentinel_files>` — Completion reporting to L1:
   - On success (all plans complete): Write `{worktree_path}/.planning/phases/{phase_dir}/STREAM_COMPLETE.md` containing stream name and list of completed plans
   - On failure: Write `{worktree_path}/.planning/phases/{phase_dir}/STREAM_FAILED.md` containing stream name, which plan failed, and reason
   - L1 detects these files to know when streams finish

7. `<constraints>` — Explicit restrictions:
   - NEVER use Bash — you do not have it
   - NEVER use Write/Edit — you do not have them (sentinel files are written by instructing L3 or via the summary output mechanism)
   - NEVER attempt TeamCreate — you are a standalone subagent (bug #32731)
   - NEVER read SUMMARY.md content — only check existence (AGNT-04)
   - Process plans sequentially within your stream — L1 handles cross-stream parallelism

Wait — correction on sentinel files: L2 has only Agent + Read. It cannot Write files directly. The sentinel file writing needs to be handled differently. Options:
- L2 spawns a minimal L3 agent to write the sentinel file
- L2 returns its completion status as its final output text, and L1 reads the agent's return value
- L1 is responsible for creating sentinel files based on L2's return

Choose this approach: L2's final output (the text it returns to L1) IS the completion signal. L2 returns a structured completion report as its final message. L1 can also check SUMMARY.md existence independently. No sentinel files written by L2.

Update the sentinel_files section to be an `<output>` section instead:

- Note: L1 reads this return value. No file writing needed by L2.
